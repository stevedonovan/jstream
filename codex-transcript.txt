› Another ergonomic improvement are trait methods which follow the pattern of getting the value of field and providing
  an explicit default, e.g. `value.get_str("boo","unknown")`. Likewise `.get_i64`, `.get_f64`, `.get_bool`. Since they
  don't return `Value` they should probably be another trait. There are also `.get_array` and `get_object`. Does this

• Yes, that feels useful, especially for the eval closure use case.

  I’d make it a separate trait, something like JsonGetExt or JsonValueExt, implemented for serde_json::Value:

  use jstream::JsonGetExt;

  let name = value.get_str("name", "unknown");
  let age = value.get_i64("age", 0);
  let active = value.get_bool("active", false);

  Suggested behavior:

  - Top-level key lookup only, matching current {field} behavior.
  - Return the explicit default if:
      - the current value is not an object,
      - the field is missing,
      - the field exists but has the wrong type,
      - the field is null.

  - get_str("boo", "unknown") should probably return &str, borrowing either the JSON string or the provided default.
  - get_i64, get_f64, get_bool return copied scalar values.
  - get_array and get_object are the only slightly tricky ones because defaults are awkward for borrowed collections.

  For arrays/objects, I’d lean toward methods that return options instead of requiring a default:

  value.get_array("items").unwrap_or(&[]);
  value.get_object("meta")

  But &[] and empty maps have lifetime/static issues unless we provide helpers/constants internally. So a pragmatic API
  could be:

  fn get_array(&self, key: &str) -> &[Value];
  fn get_object(&self, key: &str) -> Option<&Map<String, Value>>;

  Or, if you want the explicit-default pattern everywhere:

  fn get_array<'a>(&'a self, key: &str, default: &'a [Value]) -> &'a [Value];
  fn get_object<'a>(
      &'a self,
      key: &str,
      default: &'a Map<String, Value>,
  ) -> &'a Map<String, Value>;

  My recommendation:

  pub trait JsonGet {
      fn get_str<'a>(&'a self, key: &str, default: &'a str) -> &'a str;
      fn get_i64(&self, key: &str, default: i64) -> i64;
      fn get_f64(&self, key: &str, default: f64) -> f64;
      fn get_bool(&self, key: &str, default: bool) -> bool;
      fn get_array<'a>(&'a self, key: &str, default: &'a [Value]) -> &'a [Value];
      fn get_object<'a>(
          &'a self,
          key: &str,
          default: &'a serde_json::Map<String, Value>,
      ) -> &'a serde_json::Map<String, Value>;
  }

  I’d also consider making the key resolver shared with format_value, so get_str("/nested/path", "unknown") can support
  JSON Pointer too. That would make the API more consistent and immediately useful for nested ETL records.

› I'm aware that we are swallowing potential errors. One approach would be to fail early with a data schema check. A
  neat trick is to see if we can deserialize to a struct before applying any further transforms. Another way would be
  to define the schema by example, and compare fields and types, insisting that at least a certain subset of fields are
  present.


• Yes, both approaches make sense, and they solve slightly different problems.

  The deserialize to a struct approach is the strongest first layer:

  #[derive(serde::Deserialize)]
  struct Input {
      name: String,
      age: i64,
      active: bool,
  }

  serde_json::from_value::<Input>(value.clone())?;

  That gives precise Serde errors, supports nested data, optionals, renamed fields, custom deserializers, etc. The
  drawback is that it is less ad hoc: callers need to define a struct.

  The “schema by example” approach fits jstream well because it stays JSON-native:

  value
      .require(json!({
          "name": "",
          "age": 0,
          "active": true
      }))?
      .select(["name", "age", "active"])

  Possible semantics:

  - Example object defines required fields and expected JSON types.
  - Values in the example are ignored except for type.
  - Missing required fields cause an error.
  - Wrong types cause an error.
  - Extra fields in the input are allowed.
  - Nested example objects recurse.
  - Arrays are trickier; initially we could only assert “is array”, not element schema.

  For subset validation, maybe:

  .require(json!({
      "name": "",
      "age": 0
  }))

  Or a more explicit name:

  .validate_shape(json!({
      "name": "",
      "age": 0
  }))

  I’d probably add two things:

  1. validate_as::<T>()

     Consumes/returns the pipeline Value, but attempts deserialization first:

     value
         .validate_as::<Input>()?

  2. require_shape(example)

     JSON-native schema by example:

     value

             "name": "",
             "age": 0,

  type would be better than forcing schema errors into serde_json::Error, but that is probably the first point where the

  It uses the same placeholder notation as format_value:

  - {field} extracts into a top-level JSON string field.

• A few improvements feel worth considering, but I would keep them small and aligned with the current shape.

  1. Add parse_value

  parse_string currently sits outside the fluent pipeline. A parse_value/parse transform could be useful inside
  merge_eval or try_map:

  let parsed = parse_string(value.get_str("raw", ""), "{name} <{email}>")?;

  This is already possible, so this is mostly about examples rather than API.

  2. Add rename

  ETL pipelines often need field renames:

  value.rename("first_name", "given_name")
  value.rename("/person/name", "name")

  Semantics could be move-only: if source is missing, no-op; if target exists, overwrite or not? I’d default to
  overwrite for rename, because it is more like add than merge.

  3. Add copy

  Useful companion to rename:

  value.copy("name", "display_name")
  value.copy("/person/name", "name")

  This preserves source and writes target.

  4. Decide whether delete should support pointer creation semantics elsewhere

  Right now delete supports JSON Pointer paths. add does not. Eventually we may want:

  .add("/metadata/site", "corporate")

  But that changes current add semantics because a key starting with / could be intended literally. I’d postpone unless
  pointer writes become common.

  5. Add parse_string delimiter escaping or alternate delimiters

  You already called this out. I’d wait until a real string pattern forces it, but the likely API is:

  parse_string_with(text, template, Delimiters::new("[[", "]]"))

  Or simpler:

  parse_string_delimited(text, template, "[[", "]]")

  6. Split implementation modules soon

  Tests are split now, but lib.rs still contains several concerns. Before adding much more, I’d split into modules:

  src/lib.rs
  src/format.rs      // format_value, parse_string
  src/get.rs         // JsonGet
  src/stream.rs      // JsonStream
  src/shape.rs       // require_shape/select_shape helpers
  src/pointer.rs     // resolver/delete/pointer insertion helpers
  src/tests.rs

  7. Add a real error type

  We’ve intentionally postponed this, but it is probably the next “quality” step before publishing. Current
  serde_json::Error::io(...) works, but it hides domain errors. A simple crate error would improve diagnostics:

  pub enum Error {
      Json(serde_json::Error),
      Shape { path: String, expected: String, found: String },
      ParseString(String),
  }

  Then expose:

  pub type Result<T> = std::result::Result<T, Error>;

  My practical next step would be module splitting, then rename/copy. Those add real ETL value without changing the
  current mental model.
